Coding Standards & deep guidelines

A living document for everyone — from first-year students to principal engineers. Clear, consistent, and secure code starts here.

why standards matter philosophy

Code is read far more often than it is written. Consistent style reduces mental overhead, prevents bugs, and makes onboarding smoother. Standards are not about restricting creativity — they automate decisions so you can focus on solving real problems.

For beginners – they teach you proven patterns. For experts – they keep the codebase predictable across teams.

“Programs must be written for people to read, and only incidentally for machines to execute.” – Harold Abelson

naming conventions identifiers
camelCase · PascalCase · UPPER_CASE · prefixes

Names reveal intent. A reader should understand the role of a variable, function, or class without digging into implementation.

entity style examples beginner tip
local variables camelCase userName, totalAmount, isActive start with lowercase, no underscores
functions / methods camelCase (verb) fetchData(), isValid(), calculateTotal() action names
classes / interfaces PascalCase UserService, HttpRequest, Circle nouns, capital first letter
constants (immutable) UPPER_SNAKE MAX_RETRIES, API_KEY, DEFAULT_TIMEOUT all caps, underscore between words
private / internal (JS/TS) _camelCase _privateVar, _calculateHash() underscore signals “do not use outside”
booleans prefix: is, has, can, should isReady, hasPermission, canWrite reads like a question
enums / constants objects PascalCase or UPPER_SNAKE Color.RED, HttpStatus.NOT_FOUND be consistent across project
// 👍 clear intent let elapsedTimeInMs = 150; const MAX_FILE_SIZE = 1048576; class CustomerRepository { ... } // 👶 avoid: let x1, d, tmp; // what do they mean?
formatting & layout visual structure

Consistent spacing and braces make code scannable. The following rules are enforced by automated formatters (Prettier, Black, gofmt).

  • indentation: 2 spaces (no tabs) – unless language standard says otherwise (Python: 4 spaces).
  • maximum line length: 100 characters (break after operators).
  • braces: K&R style (opening brace on same line).
  • one blank line between functions, two between classes.
// ✅ good function calculateDiscount(price, rate) { if (price <= 0) { return 0; } return price * rate; } // ❌ avoid function calculateDiscount(price, rate) { if(price<=0){return 0;} return price*rate; }
comments & documentation why, not what

Explain why something is done, not what the code does (the code itself shows what). For public APIs, always include JSDoc / docstrings.

/** * Calculates the total price after tax and discount. * @param {number} basePrice - original price (>=0) * @param {number} taxPercent - 0–100 * @param {number} [discountPercent=0] - 0–100 * @returns {number} final price rounded to 2 decimals */ function computeTotal(basePrice, taxPercent, discountPercent = 0) { // tax is applied before discount (company policy) const afterTax = basePrice * (1 + taxPercent / 100); const discountMultiplier = (100 - discountPercent) / 100; return Math.round(afterTax * discountMultiplier * 100) / 100; }

Inline comments are rare – only when the code is unavoidably complex.

  • no commented-out code
  • update comments when code changes
functions & methods size & responsibility

A function should do one thing and do it well. If it exceeds 20–30 lines, consider splitting. Keep parameters minimal (0–2 ideally, use object for >3).

// 👍 focused function getActiveUsers(users) { return users.filter(u => u.active); } // 👎 too many responsibilities function processUserAndSendEmailAndLog(user) { ... } // use default parameters and destructuring function createUser({ name, email, role = 'guest' }) { ... }
error handling robustness

Never ignore errors. Use try/catch, return error objects, or promises properly. Prefer early returns to avoid deep nesting.

// ✅ safe async function fetchUser(id) { if (!id) throw new Error('user id required'); try { const response = await api.get(`/users/${id}`); return response.data; } catch (err) { logger.error({ err, id }, 'failed to fetch user'); throw new AppError('user service unavailable', 503); } } // ❌ silent fail function risky() { try { mightThrow(); } catch (e) {} // empty – never do this }
security guidelines 🔒 non‑negotiable

Even simple apps must follow these:

  • validate all input – never trust user data (sanitize, whitelist).
  • use parameterized queries – prevent SQL injection.
  • secrets out of code – environment variables or vaults.
  • keep dependencies updated – run npm audit / safety check.
  • secure headers – CSP, X-Frame-Options, etc.
// ✅ parameterized (safe) db.query('SELECT * FROM products WHERE id = $1', [productId]); // ❌ injection‑prone db.query(`SELECT * FROM products WHERE id = ${productId}`);
testing standards reliability

Write tests alongside code. Follow the arrange‑act‑assert pattern. Aim for good coverage, but focus on critical paths.

// example (jest) describe('calculateTotal', () => { it('applies tax and discount correctly', () => { // arrange const price = 100; // act const result = calculateTotal(price, 20, 10); // assert expect(result).toBe(108); // 100*1.2 =120, minus10% =108 }); });
  • unit tests
  • integration tests
  • test edge cases
git & commit hygiene history clarity

Write meaningful commit messages. Use conventional commits for automation and readability.

feat(api): add rate limiting fix(auth): handle expired tokens correctly docs(readme): update setup instructions refactor(core): split payment service test(utils): add edge cases for formatter

Branch naming: feature/, bugfix/, chore/, e.g. feature/user-profile-cache.

automation & tooling enforce standards

Let tools do the boring work. Use linters, formatters, and pre‑commit hooks.

language linter formatter
JavaScript/TS ESLint Prettier
Python flake8 / pylint black
Go golint gofmt
Java Checkstyle prettier / google-java-format

Integrate these into CI – the pipeline should fail if standards are violated.

📘 first time here?

Start with naming, formatting, and comments. Then adopt security and testing. Use the tools to automate most of it. Every expert was once a beginner — consistency comes with practice.
